title: "StartSdk facade: primary SDK entry point"
description: "StartSdk facade reference: construction, manifest threading, and helper categories from daemons and health checks to backups, interfaces, and plugins."
---

`StartSdk` is the primary entry point for service developers. It is constructed via a three-step builder chain that binds a typed manifest to the SDK, then produces an object containing every helper you need to implement the StartOS ABI. The manifest type flows through all subsequent API calls via TypeScript generics, turning typos in volume names, image IDs, dependency IDs, and plugin identifiers into compile errors rather than runtime failures.

## Construction

```typescript
import { StartSdk } from '@start9labs/start-sdk'
import { manifest } from './manifest'

export const sdk = StartSdk.of()
  .withManifest(manifest)
  .build(true)
```

<Steps>
  <Step title="StartSdk.of()">
    Creates an uninitialized `StartSdk` instance with the manifest type parameter set to `never`.
  </Step>
  <Step title=".withManifest(manifest)">
    Binds the manifest, producing a new `StartSdk<Manifest>` instance parameterized by the manifest's concrete type. From this point on, all generic constraints are derived from the manifest.
  </Step>
  <Step title=".build(true)">
    Finalizes the SDK and returns the full toolkit object. The `true` argument is a type-level gate: it only resolves to `true` (not `'Build not ready'`) when a manifest has been bound via `.withManifest()`. TypeScript enforces this at compile time.
  </Step>
</Steps>

## Type-safe manifest threading

When you call `.withManifest(manifest)`, the manifest's type is captured as a generic type parameter `Manifest extends SDKManifest`. That type then constrains every subsequent API call:

- `sdk.Mounts.of().mountVolume(name, path)` — `name` is typed as `keyof Manifest['volumes']`
- `sdk.SubContainer.of(effects, { imageId }, ...)` — `imageId` is typed as `keyof Manifest['images']`
- `sdk.checkDependencies(effects, packageIds?)` — `packageIds` elements are typed as `keyof Manifest['dependencies']`
- `sdk.plugin.url` — only non-null when `'url-v0'` appears in `Manifest['plugins']`

This propagation is automatic. You do not need to repeat the manifest type at each call site.

### Actions

Actions are user-invocable operations that appear in the StartOS UI. By convention, each action lives in its own file under an `actions/` directory.

<AccordionGroup>
  <Accordion title="sdk.Action.withInput">
    Create an action that presents a validated form to the user before executing.

    ```typescript
    sdk.Action.withInput(
      id: Id,
      metadata: MaybeFn<Omit<ActionMetadata, 'hasInput'>>,
      inputSpec: MaybeFn<InputSpecType, { effects: Effects; prefill: unknown | null }>,
      getInput: (options: { effects: Effects; prefill: DeepPartial<Input> | null }) => Promise<...>,
      run: (options: { effects: Effects; input: Input; spec: InputSpec }) => Promise<ActionResult | null>
    ): Action<Id, Input>
    ```

    Both `metadata` and `inputSpec` may be plain values or async functions receiving `{ effects }`, enabling reactive metadata (e.g. showing different descriptions based on current state).
  </Accordion>

  <Accordion title="sdk.Action.withoutInput">
    Create an action that executes immediately with no user input.

    ```typescript
    sdk.Action.withoutInput(
      id: Id,
      metadata: MaybeFn<Omit<ActionMetadata, 'hasInput'>>,
      run: (options: { effects: Effects; input: {}; spec: InputSpec }) => Promise<ActionResult | null>
    ): Action<Id, {}>
    ```
  </Accordion>

  <Accordion title="sdk.Actions">
    Build a typed action map by chaining `.addAction()` calls. Pass the result as the `actions` export from your package.

    ```typescript
    export const actions = sdk.Actions.of()
      .addAction(configAction)
      .addAction(showSecretAction)
    ```

    `Actions.of()` returns an `Actions<{}>` instance. Each `.addAction(action)` call widens the type to include the new action's ID and input type.
  </Accordion>

  <Accordion title="sdk.action.run">
    Programmatically invoke an action by ID, optionally on another package.

    ```typescript
    sdk.action.run(
      effects: Effects,
      packageId: PackageId,
      actionId: ActionId,
      input?: Record<string, unknown>
    ): Promise<ActionResult | null>
    ```
  </Accordion>

  <Accordion title="sdk.action.createTask">
    Create a task notification in the StartOS task center prompting the user to run a specific action on any package.

    ```typescript
    sdk.action.createTask(
      effects: Effects,
      packageId: PackageId,
      action: ActionInfo<ActionId, any>,
      severity: TaskSeverity,
      options?: TaskOptions<T>
    ): Promise<null>
    ```
  </Accordion>

  <Accordion title="sdk.action.createOwnTask">
    Same as `createTask`, but uses `manifest.id` as the package ID automatically — no need to pass it explicitly.

    ```typescript
    sdk.action.createOwnTask(
      effects: Effects,
      action: ActionInfo<ActionId, any>,
      severity: TaskSeverity,
      options?: TaskOptions<T>
    ): Promise<null>
    ```
  </Accordion>

  <Accordion title="sdk.action.clearTask">
    Clear one or more task notifications by their replay IDs.

    ```typescript
    sdk.action.clearTask(
      effects: Effects,
      ...replayIds: string[]
    ): Promise<null>
    ```
  </Accordion>
</AccordionGroup>

### Daemons

Daemon management controls long-running processes — their startup order, readiness probes, and graceful shutdown. `setupMain` is the `main` export for your package.

<AccordionGroup>
  <Accordion title="sdk.setupMain">
    Define the service's main entrypoint. The provided function receives `{ effects }` and must return a `Daemons` instance.

    ```typescript
    sdk.setupMain(
      fn: (o: { effects: Effects }) => Promise<Daemons<Manifest, any>>
    ): ExpectedExports.main
    ```

    ```typescript
    export const main = sdk.setupMain(async ({ effects }) =>
      sdk.Daemons.of(effects)
        .addDaemon('primary', {
          command: ['my-service', '--port', '8080'],
          mounts: sdk.Mounts.of().mountVolume('main', '/data'),
          ready: {
            fn: () => sdk.healthCheck.checkPortListening(effects, 8080),
            trigger: sdk.trigger.defaultTrigger,
          },
          requires: [],
        })
    )
    ```
  </Accordion>

  <Accordion title="sdk.Daemons.of">
    Create a new `Daemons` builder. Chain `.addDaemon()` and `.addHealthCheck()` calls to define the service topology.

    ```typescript
    sdk.Daemons.of(effects: Effects): Daemons<Manifest, never>
    ```

    The `requires` field on each daemon entry references other daemon IDs by their string name, and the builder accumulates the set of defined IDs in the type parameter to enforce that `requires` only references daemons that have been added.
  </Accordion>

  <Accordion title="sdk.Daemon.of">
    Create a single daemon definition. Use when you want to define a daemon outside of the `Daemons.of()` chain, for example in a separate file.

    ```typescript
    sdk.Daemon.of<Manifest>()
    ```

    Returns a curried function: call with `(effects, subcontainer, exec)`.
  </Accordion>
</AccordionGroup>

  </Accordion>
</AccordionGroup>

### Interfaces

Network endpoint management — bind ports and export UI, API, or P2P interfaces to users.

<AccordionGroup>
  <Accordion title="sdk.setupInterfaces">
    Define how the service is hosted and served. Runs on install, update, and whenever the user saves configuration. Returns the `setupInterfaces` export.

    ```typescript
    sdk.setupInterfaces(
      fn: (o: { effects: Effects }) => Promise<InterfaceReceipt[]>
    ): SetupInterfaces
    ```

    ```typescript
    export const setInterfaces = sdk.setupInterfaces(async ({ effects }) => {
      const multi = sdk.MultiHost.of(effects, 'web')
      const origin = await multi.bindPort(80, { protocol: 'http' })
      const ui = sdk.createInterface(effects, {
        name: 'Web UI',
        id: 'ui',
        description: 'The primary web interface.',
        type: 'ui',
        masked: false,
        schemeOverride: null,
        username: null,
        path: '',
        query: {},
      })
      return [await origin.export([ui])]
    })
    ```
  </Accordion>

  <Accordion title="sdk.MultiHost.of">
    Create a `MultiHost` instance that can bind a port and export multiple interfaces at different paths from that single port.

    ```typescript
    sdk.MultiHost.of(effects: Effects, id: string): MultiHost
    ```

    Call `multi.bindPort(internalPort, { protocol })` to get an origin, then `origin.export([...interfaces])` to publish interfaces and receive a receipt.
  </Accordion>

  <Accordion title="sdk.createInterface">
    Build a `ServiceInterface` descriptor. Pass the result to `origin.export()`.

    ```typescript
    sdk.createInterface(
      effects: Effects,
      options: {
        name: string
        id: string
        description: string
        type: 'ui' | 'api' | 'p2p'
        username: string | null
        path: string
        query: Record<string, string>
        schemeOverride: { ssl: Scheme; noSsl: Scheme } | null
        masked: boolean
      }
    ): ServiceInterfaceBuilder
    ```
  </Accordion>

  <Accordion title="sdk.serviceInterface.*">
    Helpers for reading exported interfaces at runtime.

    ```typescript
    sdk.serviceInterface.getOwn(effects, interfaceId)   // own service, by ID
    sdk.serviceInterface.get(effects, packageId, interfaceId)  // any package, by ID
    sdk.serviceInterface.getAllOwn(effects)              // all own interfaces
    sdk.serviceInterface.getAll(effects, packageId?)    // all interfaces for a package
    ```
  </Accordion>
</AccordionGroup>


    Volume and asset names are constrained to keys declared in the manifest.

### Forms

Builder-pattern system for declaring validated form inputs shown to users in the StartOS UI.

<AccordionGroup>
  <Accordion title="sdk.InputSpec.of">
    Define a form by mapping field names to `Value` instances.

    ```typescript
    sdk.InputSpec.of(spec: Record<string, Value<any>>): InputSpec<...>
    ```

    ```typescript
    const { InputSpec, Value } = sdk

    export const inputSpec = InputSpec.of({
      host: Value.text({ name: 'Host', required: true, default: 'localhost' }),
      port: Value.number({ name: 'Port', required: true, default: 8332 }),
      rpcAuth: Value.toggle({ name: 'RPC Auth', default: false }),
    })
    ```
  </Accordion>

  <Accordion title="sdk.Value">
    Factory for individual form fields. Supported types:

    | Method | Field type |
    |--------|-----------|
    | `Value.text(opts)` | Single-line text input |
    | `Value.textarea(opts)` | Multi-line text input |
    | `Value.number(opts)` | Numeric input |
    | `Value.toggle(opts)` | Boolean switch |
    | `Value.select(opts)` | Single-choice dropdown |
    | `Value.multiselect(opts)` | Multi-choice dropdown |
    | `Value.list(opts)` | Repeatable array of values |
    | `Value.color(opts)` | Color picker |
    | `Value.datetime(opts)` | Date/time picker |
    | `Value.object(opts)` | Nested sub-form |
    | `Value.union(opts)` | Conditional fields on a discriminator |
    | `Value.dynamicUnion(opts)` | Async-loaded conditional fields |
  </Accordion>

  <Accordion title="sdk.Variants">
    Build the discriminated branches used in `Value.union()` or `Value.dynamicUnion()`.

    ```typescript
    const variants = Variants.of({
      http: { name: 'HTTP', spec: InputSpec.of({ ... }) },
      https: { name: 'HTTPS', spec: InputSpec.of({ ... }) },
    })
    ```
  </Accordion>

  <Accordion title="sdk.List">
    Build list-type inputs for use with `Value.list()`.
  </Accordion>
</AccordionGroup>

### Triggers

Polling strategy functions that determine how frequently a health check fires.

<AccordionGroup>
  <Accordion title="sdk.trigger.defaultTrigger">
    Polls every 1 second while the daemon is in a pending state (`starting`, `waiting`, or `failure`). Switches to every 30 seconds once the first non-pending result is observed.

    ```typescript
    sdk.trigger.defaultTrigger: Trigger
    ```
  </Accordion>

  <Accordion title="sdk.trigger.cooldownTrigger">
    Fixed interval between health check polls.

    ```typescript
    sdk.trigger.cooldownTrigger(intervalMs: number): Trigger
    ```
  </Accordion>

  <Accordion title="sdk.trigger.statusTrigger">
    Per-status polling intervals. Map each health status to a cooldown in milliseconds. Statuses not listed fall back to 30 seconds.

    ```typescript
    sdk.trigger.statusTrigger(
      defaultMs: number,
      overrides: Partial<Record<HealthStatus, number>>
    ): Trigger
    ```

    ```typescript
    sdk.trigger.statusTrigger(30_000, {
      starting: 2_000,
      failure: 5_000,
    })
    ```
  </Accordion>
</AccordionGroup>

### Utilities

<AccordionGroup>
  <Accordion title="sdk.nullIfEmpty">

  <Accordion title="sdk.useEntrypoint">
    Signal that a daemon or subcontainer should use the container image's built-in entrypoint rather than a custom command.

    ```typescript
    sdk.useEntrypoint(overrideCmd?: string[]): UseEntrypoint
    ```

    Pass an optional `overrideCmd` array to append arguments after the entrypoint.
  </Accordion>

  <Accordion title="sdk.patterns">
    Common regex patterns and validators used in input spec validation and interface configuration (e.g. hostname format, port ranges).

    ```typescript
    sdk.patterns.hostname   // Regex for valid hostnames
    sdk.patterns.port       // Regex for valid port numbers
    // ...
    ```
  </Accordion>

  <Accordion title="sdk.setDataVersion / sdk.getDataVersion">
    Manage the data migration version during init scripts.

    ```typescript
    sdk.setDataVersion(effects: Effects, version: string | null): Promise<null>
    sdk.getDataVersion(effects: Effects): Promise<string | null>
    ```

    Call `setDataVersion` at the end of each migration step and `getDataVersion` at the start of `init` to determine which migrations still need to run.

    ```typescript
    const current = await sdk.getDataVersion(effects)
    if (!current || current < '1.1.0:0') {
      await runMigrationV1_1(effects)
      await sdk.setDataVersion(effects, '1.1.0:0')
    }
    ```
  </Accordion>
</AccordionGroup>
